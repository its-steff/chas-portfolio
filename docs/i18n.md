# Internationalization (EN/NO)

This project uses URL-based language switching plus locale-aware content from Decap CMS.

## How Language Switching Works

### 1) URL strategy

- English routes are at root:
  - `/`
  - `/gallery/`
  - `/blog/`
- Norwegian routes are under `/no/`:
  - `/no/`
  - `/no/gallery/`
  - `/no/blog/`

This gives clean, shareable URLs and better SEO than client-only text swaps.

### 2) Toggle button behavior

File: `src/assets/js/lang-toggle.js`

- Finds both nav toggle buttons via `[data-lang-toggle]`
- On click:
  - If current path starts with `/no`, removes `/no`
  - Else, adds `/no`
  - Preserves query string and hash
- Redirects to the equivalent route in the other language

### 3) Toggle label (EN/NO)

File: `src/_includes/components/nav/nav.hbs`

- Label is based on page `lang`:
  - `EN` for English pages
  - `NO` for Norwegian pages

### 4) Script + passthrough wiring

- Script is loaded in `src/_includes/layouts/base.hbs`
- JS assets are copied by Eleventy in `.eleventy.js` (`addPassthroughCopy("src/assets/js")`)

## How Home Page Localization Works

### 1) Shared template

File: `src/_includes/components/pages/home-content.hbs`

- Used by both:
  - `src/index.hbs` (`lang: en`)
  - `src/no/index.hbs` (`lang: no`)

### 2) Localized data resolver

File: `.eleventy.js`, filter: `localize`

- Resolves localized values from `src/_data/home.yml`
- Supports:
  - old flat format (single value per field)
  - Decap i18n `single_file` format (`field.en`, `field.no`)

### 3) Decap i18n setup for homepage

File: `src/admin/config.yml`

- Top-level i18n:
  - `structure: single_file`
  - `locales: [en, no]`
  - `default_locale: en`
- Homepage collection/file has i18n enabled
- Homepage fields are marked `i18n: true` (or `duplicate` for shared fields like image)

## Add Translations For More Pages (Step-by-Step)

Use this checklist for each new page/section.

### A) Create route pair

1. Create English route (if missing), e.g. `src/about.hbs`
2. Create Norwegian route, e.g. `src/no/about.hbs`
3. Set front matter:
   - EN page: `lang: en`
   - NO page: `lang: no`, `permalink: /no/about/`

### B) Move page markup into one shared partial (recommended)

1. Create partial, e.g. `src/_includes/components/pages/about-content.hbs`
2. Include it from both EN and NO route templates
3. This avoids duplicated markup and styling drift

### C) Localize data source in Decap

Choose one:

- `files` collection data (like `_data/*.yml`):
  - keep `single_file` i18n
  - mark fields `i18n: true`
- `folder` collection entries (`src/content/.../*.md`):
  - enable i18n for that collection
  - then add/edit EN + NO content entries

### D) Render localized values in template

1. For localized data objects, use `localize` helper/filter in template context
2. For static UI labels (button text, common labels), use `t` helper with `lang`
3. Prefer `@root.lang` and `@root.i18n` inside nested blocks (`with`, `each`)

### E) Make page links locale-aware

1. Ensure links from EN pages point to EN routes
2. Ensure links from NO pages point to `/no/...` routes where available
3. Keep the language toggle mapping path-to-path (`/x/` <-> `/no/x/`)

### F) Verify

1. Run `npm run dev`
2. Test EN and NO routes directly
3. Test toggle on both desktop and mobile nav
4. In CMS, switch locale and publish both EN + NO content

## Common Pitfalls

- Decap error: `i18n configuration for files collections is limited to single_file structure`
  - Fix: use `single_file` for `files` collections.

- CMS locale fields blank or missing
  - Usually field-level `i18n` is missing, or file shape does not match i18n format.

- Content disappears after adding `with` blocks
  - Nested context hides globals (`collections`, `site`, `i18n`, `lang`).
  - Use `@root.*` inside nested blocks.

- Config changes not reflected in `/admin`
  - Restart `npm run dev` and hard refresh `/admin`.
